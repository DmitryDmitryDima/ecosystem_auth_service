# Сервис Аутентификации

За что ответственен auth-сервис:
- Регистрация пользователя;
- Хранение и защита данных пользователя, таких как user uuid, password, role, username;
- Аутентификация пользователя с помощью JWT пары — access token + refresh token;
- Resolve-операции — auth сервер является единственным источником истины для пары username – uuid, в то время как данные внутри системы агрегируются по user uuid
- Поиск пользователей по username.
- Уведомление остальных компонентов системы, а также пользователей, об изменениях внутри данных или возникновении некоторых событий посредством соединения с брокером сообщений.

Используемые технологии — Spring Security, Spring Validation, Java JWT, Spring amqp (Rabbit MQ), Spring Data JPA (PostgreSQL), Spring Web.

## Хранение данных пользователей и поиск
Данные пользователей храним в таблице users. User uuid – корень для всех данных в системе, связанных с пользовательским контентом и отношениями между пользователями, их сущностями и т. д., поэтому не может быть изменен. Все остальное является мутабельным.

Набор существующих ролей — USER, ADMIN, GUEST

Так как таблица users – источник истины для всей системы, операции поиска пользователей, а также расшифровка uuid и username – обязанность auth сервера.

Для поиска пользователей по части никнейма реализован эндпоинт:
GET /auth/search?startsWith=str1&number=value1&size=value2

Он адаптирован под интерфейс Pageable, что позволяет удобно соединить его с методом в соответствующем репозитории:

```
Page<User> findByUsernameStartsWith(Pageable pageable, String startsWith);
```
Данный запрос возвращает список подходящих пар username – uuid.

Для resolve – целей имеется следующий набор endpoint’ов, также работающих с парой username - uuid:

GET /auth/resolveUUID/{uuid}
GET /auth/resolveUsername/{username}
GET /auth/resolveBatch?uuids = str

Resolve – операции при их обильном использовании могут сказаться на стабильности работы сервиса, поэтому в будущем планируется кеширование пар, с соответсвующим механизмом инвалидации записей в кеше при потере актуальности.

Также отметим наличие оптимизации под batch запросы, что полезно в ситуациях, когда фронтенд страница работает со списком, содержащим сразу несколько обезличенных uuid.

## Регистрация
POST  /auth/register
ответственен за регистрацию. Тут все максимально тривиально — пользовательский запрос содержит в себе username и password, система проверяет уникальность username, сохраняет новую сущность, шифруя пароль с помощью стандартного BCryptPasswordEncoder, отправляет ответ с полями message и success (boolean), позволяя фронтенду однозначно отобразить статус попытки регистрации, а также человекочитаемое сообщение.

В прошлой реализации также предполагалась публикация внутреннего ивента для остальных микросервисов, в частности для user service – с целью создания user properties записи, однако на данный момент решено создавать эту запись лениво, при первом обращении к странице пользователя.

Регистрация реализована примитивно, но свою функцию выполняет, пока какое-либо фундаментальное расширение не планируется.

## Аутентификация

POST /auth/login
- данный запрос является стартовой точкой в JWT (JSON Web Token) цикле, компонентами которого является access token и refresh token.

Пользователь присылает пару username – password. Происходит запрос в базу данных, где проверяется, существует ли запись с таким же username. Далее с использованием специального метода BcryptPasswordEncoder::matches проверяется присланный пароль и тот, что записан в зашифрованном виде в базе данных. В случае успеха создается пара access token – refresh token, которая вставляется как http only cookie в ответ, тем самым лишая фронтенд возможности как-то взаимодействовать с ними с помощью скриптов.

Access token создается с помощью специальной библиотеки JJWT, подпись происходит с помощью secret key, хранящегося в переменной среды.

Refresh token хранится в виде сущности в базе данных, создается из рандомной строки с помощью алгоритма хеширования (на данный момент api key в этом алгоритме отсутствует). Важный нюанс — пользователь хранит в cookie сырую строку, а ее хеш — в сущности.

Сущность refresh токена, помимо хеша токена имеет поля expired и revoked, что позволяет автоматически очищать их при потере актуальности. RefreshToken сущность имеет @ManyToOne отношение с User сущностью, что предполагает возможность существования нескольких refresh токенов для одного user.

Разберем схему работы пары access token + refresh token. Access token — короткоживущий, Refresh token — долгоживущий. Access token верифицируется при каждом запросе в защищенную часть системы. Проверяя подпись, auth сервис выполняет аутентификацию алгоритмически, без обращения к базе данных. 
Когда срок жизни access токена подходит к концу, фронтенд автоматически делает refresh запрос, обновляющий пару access token + refresh токен. При refresh запросе проверяется refresh token – его оригинальная форма хешируется, после чего происходит поиск токена по его хешу в базе данных. Если refresh токен найден, и при этом он не отозван и не просрочен, извлекается связанный с ним пользователь — на основании этих данных формируются новые токены, вставляемые в куки. Таким образом, refresh токен позволяет лучше контролировать жизненный цикл аутентификации, при этом делая операцию обновления access токена бесшовной — пользователю не нужно каждый раз логиниться при просрочке access токена. За счет того, что access токен живет недолго, с помощью refresh токена мы также можем реализовать logout, попросту отозвав все refresh токены, связанные с пользователем.

Выполнив refresh, фронтенд либо повторяет запрос, либо в случае неудачи с обновлением пары перенаправляет пользователя на /login эндпоинт.

## Валидация доступа

GET /auth/validate

К данному эндпоинту обращается API gateway каждый раз, когда пользователь пытается получить доступ к защищенной части системы. Запрос должен содержать в себе куки с access токеном и @RequestHeader с targetUsername, если имеет место ситуация, где некий viewer просматривает контент некоего target, используя username.

Ответ приходит в форме ValidationResponse {uuid, targetUUID, username, role}, что впоследствии используется как Security Context для внутренних сервисов, избавляя последние от необходимостb заниматься аутентификацией.

Таким образом, эндпоинт валидирует access токен и автоматизирует resolve операцию для target username, позволяя сформировать полноценный и понятный security контекст для адресата изначального запроса.

Если токена нет — то отправляется пустой ответ с role = GUEST, если токен есть, но он просрочен или некорректен — возвращается 401, что на стороне фронтенда провоцирует refresh запрос.

## Refresh запрос
POST /auth/refresh

В данном случае чтение и запись информации происходит только на уровне cookie, поэтому запрос ничего не несет с собой и не имеет какого-либо ответа.

Читается хранимый в http-only cookie refresh token (его строковая, raw форма).

Raw форма хешируется (алгоритм хеширования должен содержать secret key), после чего происходит поиск в базе данных записи с сформированным хешем. Если такая запись есть, и она не просрочена и не отозвана, то извлекается пользователь, ассоциированный с данным токеном (@ManyToOne связь), после чего на основе извлеченных данных происходит генерация новой пары токенов. Они помещаются в cookies ответа.

## Logout
POST /auth/revoke  - пока что logout максимально прост — мы извлекаем raw форму из кук, ищем хеш в бд, ставим revoke. В теории это можно расширить до поиска всех ассоциированных с пользователем refresh токенов, внедряя таким образом logout со всех устройств.
